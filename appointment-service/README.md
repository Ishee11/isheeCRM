# appointment-service CI/CD

## Runtime Model

GitHub Actions builds the application image and pushes it to GHCR with two tags:

- commit SHA, for example `c301a2d`
- `main`

The production server does not need the repository source code. It keeps only a
small deploy directory with:

- `.env` with runtime secrets and port settings
- `docker-compose.yml` generated by the workflow

## Current Server Ports

The current production host also runs the poker project:

- poker prod app: `0.0.0.0:8080`
- poker prod PostgreSQL: `127.0.0.1:5432`
- poker dev app: `0.0.0.0:18080`
- poker dev PostgreSQL: `127.0.0.1:15432`

Until `crm.semenovv.space` is configured in DNS and nginx, appointment-service
uses:

- app: `0.0.0.0:8081`
- PostgreSQL: `127.0.0.1:5433`

Nginx currently proxies `semenovv.space` to poker on `127.0.0.1:8080`, so
appointment-service should be exposed through a separate host name such as
`crm.semenovv.space`.

## Server Setup

Create a deploy directory on the server:

```bash
mkdir -p /opt/appointment-service
cd /opt/appointment-service
```

Create `/opt/appointment-service/.env`:

```env
APP_PORT=8081
APP_CONTAINER_NAME=appointment-service-app-container
APP_IMAGE=ghcr.io/ishee11/isheecrm-appointment-service
IMAGE_TAG=latest

DB_HOST=postgres
DB_PORT=5432
DB_USER=postgres
DB_PASSWORD=change-me
DB_NAME=isheecrm

POSTGRES_IMAGE=postgres:17.6-bookworm
POSTGRES_CONTAINER_NAME=postgres-container
POSTGRES_PORT=5433
POSTGRES_BIND_ADDRESS=127.0.0.1
POSTGRES_USER=postgres
POSTGRES_PASSWORD=change-me
POSTGRES_DB=isheecrm
```

Use real values for `DB_PASSWORD` and `POSTGRES_PASSWORD`. If the app connects
to the PostgreSQL container from this compose project, `DB_USER`,
`DB_PASSWORD`, and `DB_NAME` should match `POSTGRES_USER`, `POSTGRES_PASSWORD`,
and `POSTGRES_DB`.

The workflow updates only `APP_IMAGE` and `IMAGE_TAG` in `.env`. Database
credentials stay on the server and are not stored in GitHub Secrets.

## GitHub Actions Secrets

Create these repository secrets in GitHub:

- `SSH_HOST`
- `SSH_PORT`
- `SSH_USER`
- `SSH_PRIVATE_KEY`
- `DEPLOY_PATH`
- `REGISTRY_USERNAME`
- `REGISTRY_PASSWORD`

`DEPLOY_PATH` is the server deploy directory, for example:

```text
/opt/appointment-service
```

`REGISTRY_USERNAME` is usually your GitHub username. `REGISTRY_PASSWORD` should
be a GitHub PAT that can read GHCR packages, typically with `read:packages`.

`GITHUB_TOKEN` does not need to be created manually; GitHub Actions provides it
for the build-and-push job.

## Deployment Flow

On push to `main`, or when the workflow is run manually, GitHub Actions:

1. Runs `go test ./...` in `appointment-service`.
2. Builds the Docker image.
3. Pushes the image to GHCR.
4. Connects to the server over SSH.
5. Creates `DEPLOY_PATH` if needed.
6. Generates `docker-compose.yml` in `DEPLOY_PATH`.
7. Updates `APP_IMAGE` and `IMAGE_TAG` in `.env`.
8. Runs:

```bash
docker compose --env-file .env -p appointment-service pull
docker compose --env-file .env -p appointment-service up -d --remove-orphans
```

If `.env` is missing, the workflow writes `.env.example` into `DEPLOY_PATH` and
fails with an explicit message. Fill `.env` on the server and rerun the
workflow.

## Nginx Example

The temporary direct URL is:

```text
http://193.238.134.58:8081
```

After `crm.semenovv.space` is configured in DNS, switch the Docker port binding
back to localhost and use nginx to expose the app publicly:

```nginx
server {
    server_name crm.semenovv.space;

    location / {
        proxy_pass http://127.0.0.1:8081;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

Then issue a certificate with Certbot for `crm.semenovv.space`.

## Local Development

Run the service locally from this directory:

```bash
go run ./cmd/appointment-service
```

Run tests:

```bash
go test ./...
```

Run local Docker Compose:

```bash
docker compose --env-file .env up -d
```

## Notes

- Keep production secrets only in `.env` on the server.
- The deploy workflow no longer runs `git pull` on the server.
- The server does not need `cmd/`, `internal/`, `go.mod`, migrations, or other
  source files for normal deploys.
- PostgreSQL data is stored in the Docker volume
  `appointment-service_isheecrm_isheecrm_data` when using project name
  `appointment-service`.
